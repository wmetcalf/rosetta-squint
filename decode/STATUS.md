# rosetta-image-decode — Status & Setup

Byte-exact, PIL-compatible image decoders across **Java, Go, Rust, JavaScript/TypeScript, and Swift**.

Sibling project to [rosetta-image-hash](../rosetta-image-hash). That library hashes pre-decoded RGB buffers; this one produces those buffers from arbitrary image bytes, with cross-language pixel-level parity.

---

## Supported formats

| Format | Reference lib (decoder) | Rust | Go | JS | Swift | Java | Tag |
|---|---|---|---|---|---|---|---|
| BMP | Pillow built-in | ✓ byte-exact | ✓ | ✓ | ✓ | ✓ | `bmp-v0.1.0` |
| PNG | Pillow built-in | ✓ | ✓ | ✓ | ✓ | ✓ | `png-v0.1.0` |
| GIF (first frame) | Pillow built-in | ✓ | ✓ | ✓ | ✓ | ✓ | `gif-v0.1.0` |
| JPEG | libjpeg-turbo (JDCT_ISLOW) | ✓ | ✓ | ✓ | ✓ | ✓ | `jpeg-v0.1.0` |
| WebP | libwebp | ✓ | ✓ | ✓ | ✓ | ✓ | `webp-v0.1.0` |
| TIFF (uncompressed / LZW / Deflate, RGB+grayscale, baseline) | libtiff via PIL | ✓ | ✓ | ✓ | ✓ | ✓ | `tiff-v0.1.0` |
| HEIC (HEVC, single still image) | system libheif 1.17.6 | ✓ byte-exact | ✓ byte-exact | ⚠ ±2 px tolerance | ✓ byte-exact | ✓ byte-exact | `heic-v0.1.0` |

Shared core: `spec-decode-v0.1.0`.

**Out of scope for v1:** EMF/WMF (dropped), AVIF, animated WebP iteration, multi-page/multi-image TIFF, HEIC bursts/grids/depth/HDR, color-managed pipelines (no ICC profile honoring), metadata extraction.

---

## Why HEIC has a tolerance and nothing else does

libheif's WASM build (`libheif-js`, bundled with the JS port) diverges from the system Linux build by **±1–2 per pixel** on lossy fixtures, due to different YCbCr→RGB rounding paths. Goldens are anchored to system libheif 1.17.6 (so Rust/Go/Java/Swift remain byte-exact); the JS port runs a max-delta tolerance for HEIC only. Documented in `js/rosetta-image-decode/DECODER_NOTES.md`. Every other format on every other port is strict byte-exact.

---

## How "tested against Python" actually works

Goldens are generated by `spec/regenerate.py`, which runs `PIL.Image.open(fixture).tobytes()` for every supported format and freezes the raw RGB(A) bytes into `spec/decoded/<format>/<fixture>.bin`. The SHA256 of each buffer is also recorded in `spec/goldens.json`. Each port's Group 2 test decodes the same fixture and asserts byte-for-byte equality with the frozen buffer.

So a Group 2 pass means: port output equals what PIL 11.0.0 produced at fixture-gen time.

Two exceptions worth knowing:
- **HEIC goldens** are not pillow-heif output. PIL's `pillow-heif` plugin bundles libheif 1.21.2; system libheif is 1.17.6; the two differ by ±1 px. We regenerated HEIC goldens via a ctypes wrapper around system libheif 1.17.6 (see `spec/SPEC.md` §16). pillow-heif is still used to *encode* the HEIC fixtures in `spec/synth_heic.py`, just not to decode them for goldens.
- **PIL reference is frozen at fixture-gen time.** No test runs PIL live alongside the port and diffs them. If Pillow ships a behavior change in a future version, our goldens go stale until someone re-runs `regenerate.py --check`.

## Installation & build per language

Each port has its own FFI strategy for JPEG/WebP/TIFF/HEIC, which drives the install steps below.

### Common system dependencies (Linux, Ubuntu 24.04)

These are shared by Java, Go, Rust, and Swift. JS bundles everything as WASM and needs none of them.

```bash
sudo apt-get install -y \
    libjpeg-turbo8-dev libturbojpeg libturbojpeg0-dev libturbojpeg-java \
    libwebp-dev libwebp7 libsharpyuv-dev \
    libtiff-dev \
    libheif-dev libheif-plugin-libde265 libde265-dev libde265-0 \
    pkg-config
```

For macOS Homebrew (untested):

```bash
brew install jpeg-turbo webp libtiff libheif pkg-config
```

The Java port additionally expects the system TurboJPEG JAR at `/usr/share/java/turbojpeg.jar` (provided by `libturbojpeg-java` on Ubuntu/Debian).

---

### Rust

```bash
cd rust/rosetta-image-decode
cargo build
cargo test
```

Crate dependencies:
- `image = "0.25"` (PNG, GIF, TIFF features — pure-Rust decoders)
- `mozjpeg-sys = "2"` — compiles libjpeg-turbo from source; **no system libjpeg needed**
- `libwebp-sys2 = "0.1.11"` — compiles libwebp from source; **no system libwebp needed**
- `libheif-rs = "2.7"` with feature `v1_17` — links to **system libheif** via pkg-config (so libheif-dev IS required)
- `thiserror = "1"`

So Rust needs `libheif-dev` + `libde265` system-installed but compiles JPEG/WebP itself.

### Go

```bash
cd go/imagedecode
go test -count=1 ./...
```

Module deps:
- `github.com/pixiv/go-libjpeg` — cgo binding to system libjpeg-turbo
- `github.com/chai2010/webp` — cgo binding to system libwebp
- `github.com/strukturag/libheif v1.17.6` (pin must match system libheif version — earlier versions of the bindings reference symbols missing in libheif 1.17)
- `golang.org/x/image/tiff` — pure Go, no system lib

Needs `libjpeg-turbo8-dev`, `libturbojpeg0-dev`, `libwebp-dev`, `libheif-dev`, `pkg-config`.

### Java (Maven)

```bash
cd java/rosetta-image-decode
mvn -B -ntp test
```

Requires JDK 17+. Maven coordinates:
- `org.libjpeg-turbo:turbojpeg:2.1.5` — system-scope dependency at `/usr/share/java/turbojpeg.jar` (Ubuntu pkg `libturbojpeg-java`)
- `org.sejda.imageio:webp-imageio:0.1.6` — pure-Java WebP plugin (bundles native libwebp loader)
- `com.twelvemonkeys.imageio:imageio-tiff:3.10.1` — TIFF plugin
- `net.java.dev.jna:jna:5.18.1` — used by our hand-written libheif JNA wrapper in `io.rosetta.imagedecode.internal.libheif`

PNG and GIF use `javax.imageio` (stdlib). The HEIC JNA wrapper dynamically loads `libheif.so.1` (Linux) or `libheif.dylib` (macOS); `libheif-dev` must be installed.

### JavaScript / TypeScript

```bash
cd js/rosetta-image-decode
npm install
npm test
```

Requires Node 18+. **No system libraries needed** — everything bundles natively or as WASM:

- `pngjs ^7` — pure-JS PNG
- `omggif ^1` — pure-JS GIF
- `utif2 ^4.1.0` — pure-JS TIFF (CommonJS, imported via `createRequire` in our ESM context)
- `@jsquash/jpeg ^1.6.0` — mozjpeg compiled to WASM
- `@jsquash/webp ^1.5.0` — libwebp compiled to WASM
- `libheif-js ^1.17.1` — libheif + libde265 compiled to WASM (this is the one that diverges ±1 px from system libheif; see top of doc)

BMP is implemented inline in `src/internal/bmp.ts` (no library).

`decode()` is async because the WASM modules initialize asynchronously. All Group 2 tests `await decode(...)`.

### Swift

```bash
cd swift/RosettaImageDecode
swift build
swift test
```

Requires Swift 5.9+. Linux dev path used here: `/opt/swift/swift-5.9.2-RELEASE-ubuntu22.04/usr/bin` (export it on `$PATH` before invoking `swift`).

Package.swift declares 4 `systemLibrary` targets pulling C libraries via `pkg-config`:

| Module | pkg-config | Linux apt | macOS Homebrew |
|---|---|---|---|
| `Cjpeg` | `libturbojpeg` | `libturbojpeg0-dev` | `jpeg-turbo` |
| `Cwebp` | `libwebp` | `libwebp-dev` | `webp` |
| `Ctiff` | `libtiff-4` | `libtiff-dev` | `libtiff` |
| `Cheif` | `libheif` | `libheif-dev` | `libheif` |

Plus one SwiftPM dep: [`tayloraswift/swift-png`](https://github.com/tayloraswift/swift-png) for PNG. GIF has a hand-written pure-Swift decoder in `Sources/RosettaImageDecode/Internal/GIFDecoder.swift` (LZW + 4-pass interlacing; ~518 LOC).

**macOS:** *should* work — `.brew([...])` providers are declared on every systemLibrary, the C APIs are identical to Linux, Swift 5.9 ships with Xcode 15. But none of the per-port `swift test` runs have been executed on macOS. Most likely first failure on Mac would be library link names if Homebrew puts a keg-only formula in a non-default `pkg-config` search path; resolve with `PKG_CONFIG_PATH=$(brew --prefix libheif)/lib/pkgconfig swift test` etc.

## Regenerating goldens

```bash
cd spec
python3 -m venv .venv && source .venv/bin/activate
pip install Pillow pillow-heif jsonschema
python3 regenerate.py             # writes spec/decoded/**/*.bin + updates goldens.json
python3 regenerate.py --check     # exits 1 if outputs would differ from committed (excluding timestamp)
python3 consistency.py            # schema validation + counts
```

**Important:** `regenerate.py --format <fmt>` is destructive — it rewrites `goldens.json` with only that format's entries. To regenerate a single format without nuking the others, use the merge pattern from `spec/SPEC.md` (or just `git checkout spec/goldens.json` after to restore the rest).

HEIC goldens require an extra step because system libheif (not pillow-heif) is the reference — see the ctypes wrapper script and notes in `spec/SPEC.md` §16.

## Known gaps & caveats

- **No macOS testing.** All test runs are Linux x86-64. Swift on Mac is plausible-but-unverified.
- **No CI execution.** `.github/workflows/ci.yml` exists with proper apt-install steps per port matrix entry, but the repo is local-only and has never been pushed to GitHub.
- **No live Python diff job.** Goldens are frozen at fixture-gen time; nothing runs PIL and a port side-by-side in the same process.
- **No cross-port end-to-end diff job.** Cross-port equivalence is *implied* by every port matching the same goldens, but no test runs all 5 ports on the same input and diffs the outputs directly.
- **HEIC has a JS tolerance** — see top of doc.
- **No published packages.** Nothing on crates.io / Maven Central / npm / Swift Package Index.

## Repository layout

```
rosetta-image-decode/
├── spec/                                   # shared bit-level specification + golden generator
│   ├── SPEC.md                             # §1-9 shared core, §10-16 per format
│   ├── formats.json                        # status, reference libs, ports done per format
│   ├── formats.schema.json                 # JSON schema for formats.json
│   ├── errors.json                         # per-invalid-fixture expected (kind, format, detail)
│   ├── goldens.json                        # SHA256 + width/height/channels per fixture
│   ├── regenerate.py                       # PIL-driven golden regenerator
│   ├── consistency.py                      # validate schemas + cross-references
│   ├── synth_{bmp,jpeg,webp,tiff,heic}.py  # synth fixture generators
│   ├── fixtures/{bmp,png,gif,jpeg,webp,tiff,heic}/{valid,invalid}/
│   └── decoded/{bmp,png,gif,jpeg,webp,tiff,heic}/valid/*.bin

└── swift/RosettaImageDecode/
```
